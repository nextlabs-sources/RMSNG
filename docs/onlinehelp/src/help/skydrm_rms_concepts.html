SkyDRM RMS concepts

This section describes the SkyDRM RMS-specific terminology and components.

SkyDRM RMS deployment package consists of four components, which are delivered as Docker containers.
  • RMS: The web server application which provides the business logic and web applications to perform the following functions.
    • User management
    • User authentication
    • Key management
    • Projects
    • MySpace
    • Document classification
    • Policy creation
    • Administration
  • Router: A web application which handles the routing of client requests to the server.
  • Viewer: A web application which renders files for viewing.
  • Database: The database in which all SkyDRM RMS data is stored.
  • CacheServer: A distributed data storage that caches data within the Docker swarm network.
  • MessageQueue: A web service that sends and receives notifications. In SkyDRM RMS, the message queue service uses RabbitMQ.

Administrator rights

Administrator rights enable users to reclassify files without needing to access the token used to protect the file. This is a key point, because there may not be a policy in place which allows access to the administrators who needs to reclassify the files. The roles with administrator rights are:

  • All the project administrators: the person who created the project, and any person with administrator rights who are members of the project
  • Tenant administrators
  • API users

API user

An API user is a privileged role which can obtain an appId and appKey from the super administrator. You need the appId and appKey to use the SkyDRM Server SDK.

The API user can also perform the following tasks:
  • listing projects
  • creating projects
  • encrypting and decrypting files on different tenants
  • partially decrypting files
  • updating tags

Policies deployed at the domain, tenant, or project level do not apply to API users, which means that an API user can reclassify files without being a member of the project.

System bucket

In SkyDRM, a project issues the token that encrypts and protects a file. However, in some enterprises and usage scenario, projects are not required. For example, when you want to protect a file in your local drive, or in a network drive, you should not be required to upload the file to a project first. To bypass projects, you can use the system bucket, which issues encryption tokens for a file without having to upload the file to a project.
